# Azure Monitor Ops — live KQL triage

Run a specific KQL query against **real** telemetry through the **az MCP**: query Azure Monitor / Application Insights / Log Analytics (and ADX via Kusto when the profile allows), pull metrics and the activity log, and hand back a reproducible result set. The skill is **advisory** — it QUERIES and PROPOSES, it does not create alerts, change config, or remediate. Every row in the output comes from an actual `mcp__azure__monitor` / `mcp__azure__kusto` result; nothing is invented from memory, and the **exact KQL executed is always printed** so a human can re-run it.

**Read-only is NOT a property this skill can enforce.** A skill has no `tools:` allowlist, so it cannot restrict which MCP tools the model calls — the `mcp__azure__monitor` namespace itself is NOT read-only (it contains `monitor_webtest_create-or-update` and Workbooks create/update/delete, all `Destructive:true`). The *only* real read-only control is launching the Azure MCP server with the `--read-only` flag (couche 0 — skills advise, infra enforces). Operators MUST start the az MCP with `--read-only`; without that flag, nothing here stops a write call. The deterministic side likewise lives elsewhere: `observability-gen` defines the signals and emits the code, and the gate that stops a bad deploy is the CI/branch-protection pipeline + alert rules — not this skill.

## When to use
- "run this KQL", "query our logs", "list the tables", "check App Insights", "query Log Analytics", "what does the activity log show" — explicit query verbs only
- You have a specific query to run and need to READ live telemetry / metrics / logs / activity events for a named symptom — NOT "why is it down" open-ended incident triage (that is `azure-sre`)
- A runbook step (from `incident-runbook`) says "pull the error signature from Log Analytics" — this skill is that live-query arm
- An `azure-sre` triage needs a raw single-KQL pull as one input — this skill is that arm; it does not own the incident

## When NOT to use
- "Why is prod down / erroring / slow", "triage this incident", "our fault or an Azure platform event?" → `azure-sre` (it OWNS open-ended live incident triage — drives `mcp__azure__applens` / `mcp__azure__sreagent` / `mcp__azure__resourcehealth`, root-causes, and proposes ranked remediation, calling THIS skill's raw single-KQL pull as one input). The boundary is verb-explicit: a named query verb is this skill; an open "why is it broken" is `azure-sre`.
- Generating OTel SDK init, semantic conventions, or the SLI/SLO + alert-rule *definitions* → `observability-gen` (it EMITS code on the generate side of the generate-vs-operate axis; this skill only QUERIES live data, it never instruments)
- Defining the error-budget burn policy / its delivery gate → `error-budget`
- Writing the incident plan / recovery playbook → `incident-runbook` (this skill is the live-query arm a runbook step invokes, not the plan)
- `obs_backend` is not `azure-monitor` (or, for the Kusto/ADX path, not `adx`), or no live workspace/cluster is reachable → refuse (do not fabricate query results)

## Inputs
- **az MCP — launched with `--read-only`** — the query is built entirely from these tools. The `--read-only` server flag is what keeps the call read-only; the `monitor` namespace also carries write tools (webtest/workbook create-update-delete) that the flag blocks. Use only the read tools below:
  - `mcp__azure__monitor` → run KQL over the Log Analytics workspace + resource logs, pull metrics, and read the activity log
  - `mcp__azure__applicationinsights` → list the App Insights component(s) backing the service and resolve which workspace its telemetry lands in
  - `mcp__azure__kusto` → run KQL against Azure Data Explorer (ADX), ONLY when `obs_backend` is `adx` (a separate backend with its own cluster — NOT implied by `azure-monitor`)
- `org-profile.yaml` → `obs_backend` — REQUIRED. `azure-monitor` enables the Log Analytics / App Insights path (`mcp__azure__monitor`); `adx` enables the Kusto path (`mcp__azure__kusto`). Any other value = wrong backend → refuse and route to the matching ops skill. A workspace on `azure-monitor` does NOT imply an ADX cluster, so the Kusto path is gated separately.
- `org-profile.yaml` → `platform` — scopes which resource types / tables to focus the query on (aks → container/pod logs; webapp → App Service logs + App Insights requests; serverless → Functions traces)
- `stack-interface.yaml` (OPTIONAL) → when present, resolve workspace / login / subscription context through the `stack-adapter` agent; never emit raw `az login` / `az account set`

## Steps
1. **Confirm `--read-only` + a live workspace — never hallucinate results.** Assume the operator launched the az MCP with `--read-only`; if you cannot confirm that, say so — without it the read-only posture is unenforced. Read `org-profile.yaml`; if `obs_backend` is neither `azure-monitor` nor `adx`, REFUSE and name the matching backend's ops path. For `azure-monitor`, resolve the App Insights component(s) with `mcp__azure__applicationinsights` and confirm the backing Log Analytics workspace answers; for `adx`, confirm the ADX cluster answers. If nothing resolves (az MCP not connected / no auth / no workspace or cluster), REFUSE — do not invent rows. If `stack-interface.yaml` exists, get the login/workspace context from `stack-adapter` first (embed verbatim; on `MISSING-MAPPING`, stop).
2. **Identify the workspace + tables.** List the available tables for the workspace via `mcp__azure__monitor` (e.g. `AppRequests`, `AppExceptions`, `AppTraces`, `ContainerLogV2`, `AzureActivity`) and pick the ones relevant to the triage question (errors, latency, saturation, recent deploys). Scope by `org-profile.yaml` `platform`.
3. **Compose the KQL for the triage question.** Write explicit, time-bounded KQL (always `| where TimeGenerated > ago(<window>)`) for the symptom: top error signatures (`AppExceptions | summarize count() by problemId, outerMessage`), latency (`AppRequests | summarize percentiles(DurationMs, 50, 95, 99)`), failure rate, or saturation. Keep each query small and named so it is reproducible.
4. **Run the KQL via the az MCP (read tools only).** Execute against Log Analytics with `mcp__azure__monitor` when `obs_backend == azure-monitor`, or against ADX with `mcp__azure__kusto` when `obs_backend == adx`. Pull correlating **metrics** (`mcp__azure__monitor` metrics) and the **activity log** (`AzureActivity` / activity-log query) so a spike can be time-correlated to a deploy, scale event, or config change. Call only the read tools — never `monitor_webtest_create-or-update`, workbook create/update/delete, or any other write tool in the namespace. Capture the real result set — no estimation.
5. **Return the result set + a reproducible triage summary.** Emit the top error signatures with counts, the latency/saturation profile, and the time correlation to deploys/changes — and ALWAYS print **the exact KQL executed** (and the workspace id / time window) for each query so the run is reproducible in the portal. Use the Output contract below.
6. **State the boundary, loudly.** Close by asserting: this is a read-only query — it READS the signals `observability-gen` defined and that a runbook step or `azure-sre` can invoke; it created no alerts, changed no config, and remediated nothing. State plainly that the skill itself cannot enforce read-only — the `--read-only` az MCP launch flag is the only control — so confirm that flag was set. Open-ended incident triage + remediation orchestration is `azure-sre` (applens/sreagent/resourcehealth); the deterministic gate is the CI/branch-protection pipeline + the alert rules, not this skill. List exactly which az MCP read tools were used.

## Output / validation
- A result set with: top error signatures (signature + count + sample message), a latency/saturation profile, the activity-log/deploy correlation, AND **the exact KQL executed** for every query, plus the workspace/cluster id and time window — sourced entirely from `mcp__azure__monitor` (azure-monitor) / `mcp__azure__kusto` (adx), no invented rows.
- Validate: re-running the printed KQL in the Azure portal (Log Analytics for `azure-monitor`, ADX for `adx`) against the same workspace/cluster + window yields the same result shape; every cited count came from a real az MCP call. No placeholder numbers, no from-memory error messages.
- Boundary reiterated: this skill **queries**, it does not generate or enforce, and it cannot itself enforce read-only — the `--read-only` az MCP launch flag is the control. `observability-gen` emits the OTel init + SLO/alert definitions (generate side); `azure-sre` owns open-ended triage + root-cause + remediation; the deterministic gate is the pipeline + alert rules. This skill blocks and changes nothing.

## Refuses when
- `obs_backend` in `org-profile.yaml` is neither `azure-monitor` (Log Analytics / App Insights path) nor `adx` (Kusto path) — wrong backend; refuse and route to the matching backend's ops path. Do NOT run the Kusto path off an `azure-monitor` value — ADX is a distinct backend with its own cluster.
- No live workspace/cluster is reachable via the az MCP (`mcp__azure__applicationinsights` / `mcp__azure__monitor` / `mcp__azure__kusto` not connected / unauthenticated / no workspace or cluster) — refuse rather than fabricate query results, error signatures, or counts.
- `stack-interface.yaml` is present and `stack-adapter` returns `MISSING-MAPPING` for the login/workspace op — stop; do not fall back to a raw `az` command.
- Asked for open-ended "why is prod down/erroring/slow" incident triage, root-cause, or remediation — out of scope; route to `azure-sre`. This skill runs a named query and returns rows; it does not orchestrate an incident.
- Asked to MUTATE anything (create an alert rule, a webtest, a workbook, change a diagnostic setting, set a metric threshold, remediate) — out of scope. Note that this skill cannot itself block such a call (no `tools:` allowlist) — the `--read-only` az MCP launch flag is what blocks writes. Route alert/SLO *definition* to `observability-gen` and remediation orchestration to `azure-sre`.
- Asked to GENERATE OTel init / SLI-SLO / alert code (`observability-gen`), define error-budget burn policy (`error-budget`), or be presented as a hard gate that blocks merges — route to the named sibling; this skill only queries and advises.
